# Multi-Tenant SaaS — Design

**Spec:** `.specs/features/multi-tenant-saas/spec.md`
**Status:** Draft

---

## Architecture Overview

One shared API deployment. Per-client frontend deployments. All data in one DB, scoped by `tenantId` row-level. Tenant resolved from `X-Tenant-Slug` header on every request.

```
┌─────────────────────────────────────────────────────┐
│                  Coolify VPS                         │
│                                                      │
│  [soberano.altion.com.br]  ──►  web-soberano        │
│  [marques.altion.com.br]   ──►  web-marques    ─┐   │
│  [admin.altion.com.br]     ──►  web-admin      ─┤   │
│                                                  │   │
│  [api.altion.com.br]  ◄─────────────────────────┘   │
│       │                                              │
│  [PostgreSQL]  ◄────── single DB, row-level tenancy  │
└─────────────────────────────────────────────────────┘
```

**Request lifecycle (frontend → API):**
```
Frontend sets X-Tenant-Slug: soberano on every request
  → tenantMiddleware resolves Tenant row
  → createTenantPrisma(tenant.id) creates scoped Prisma client
  → request.tenantPrisma attached to request
  → route handler creates repositories with request.tenantPrisma
  → all queries automatically scoped to tenantId
```

## Components

### 1. Prisma Schema — New Models + Renames

**Purpose:** Add `Tenant` model, rename `Barber` → `Provider`, add `tenantId` FK everywhere.

**Location:** `packages/api/prisma/schema.prisma`

**New Tenant model:**
```prisma
model Tenant {
  id          String   @id @default(uuid()) @db.Uuid
  slug        String   @unique @db.VarChar(50)
  name        String   @db.VarChar(200)
  type        String   @default("barbershop") @db.VarChar(50)
  config      Json     @default("{}")
  isActive    Boolean  @default(true) @map("is_active")
  createdAt   DateTime @default(now()) @map("created_at") @db.Timestamptz

  providers    Provider[]
  providerShifts ProviderShift[]
  providerAbsences ProviderAbsence[]
  services     Service[]
  customers    Customer[]
  appointments Appointment[]

  @@map("tenants")
}
```

**Rename pattern (Barber → Provider):**
- `@@map("barbers")` → `@@map("providers")`
- `@map("barber_id")` → `@map("provider_id")`
- `@map("barber_shifts")` → `@map("provider_shifts")`
- `@map("barber_absences")` → `@map("provider_absences")`
- `slug @unique` → `@@unique([tenantId, slug])` (slug unique per tenant, not globally)
- `email @unique` → `@@unique([tenantId, email])`

**tenantId added to:** `Provider`, `ProviderShift`, `ProviderAbsence`, `Service`, `Customer`, `Appointment`

**HTTP route paths stay unchanged:** `/api/barbers`, `/api/barbers/:id` — internal rename only, no frontend breakage.

**Response JSON keys stay unchanged:** The routes still return `{ barbers: [...] }`. Frontend needs zero changes for the rename.

### 2. Migration Strategy (Zero Downtime)

**Purpose:** Safely migrate Soberano's production data to the new schema.

**Location:** `packages/api/prisma/migrations/`

**Three sequential migrations:**

**Migration 1 — Add tenants, nullable tenant_id:**
```sql
-- Auto-generated by Prisma for Tenant model + nullable columns
-- Manually append:
INSERT INTO tenants (id, slug, name, type, config, is_active, created_at)
VALUES (
  gen_random_uuid(),
  'soberano',
  'Soberano Barbearia',
  'barbershop',
  '{"businessName":"Soberano Barbearia","providerLabel":"Barbeiro","bookingUrl":"https://soberano.altion.com.br"}',
  true,
  NOW()
);

-- Assign all existing rows to Soberano tenant
UPDATE providers   SET tenant_id = (SELECT id FROM tenants WHERE slug = 'soberano');
UPDATE services    SET tenant_id = (SELECT id FROM tenants WHERE slug = 'soberano');
UPDATE customers   SET tenant_id = (SELECT id FROM tenants WHERE slug = 'soberano');
UPDATE appointments SET tenant_id = (SELECT id FROM tenants WHERE slug = 'soberano');
-- (also provider_shifts, provider_absences)
```

**Migration 2 — Make tenant_id NOT NULL + indexes:**
```sql
ALTER TABLE providers    ALTER COLUMN tenant_id SET NOT NULL;
ALTER TABLE services     ALTER COLUMN tenant_id SET NOT NULL;
ALTER TABLE customers    ALTER COLUMN tenant_id SET NOT NULL;
ALTER TABLE appointments ALTER COLUMN tenant_id SET NOT NULL;
-- Add performance indexes
CREATE INDEX ON appointments (tenant_id, date);
CREATE INDEX ON providers (tenant_id);
CREATE INDEX ON customers (tenant_id);
```

**Migration 3 — Rename barbers → providers:**
```sql
ALTER TABLE barbers         RENAME TO providers;
ALTER TABLE barber_shifts   RENAME TO provider_shifts;
ALTER TABLE barber_absences RENAME TO provider_absences;
-- Rename FK columns
ALTER TABLE provider_shifts   RENAME COLUMN barber_id TO provider_id;
ALTER TABLE provider_absences RENAME COLUMN barber_id TO provider_id;
ALTER TABLE appointments      RENAME COLUMN barber_id TO provider_id;
```

> **Why 3 migrations?** Splitting allows rollback at each step. Migrations 1 and 2 are fully reversible. Migration 3 (rename) can run after verifying production is stable.

### 3. `createTenantPrisma` Factory

**Purpose:** Returns a Prisma client extended to auto-scope all queries to a tenantId.

**Location:** `packages/api/src/config/tenant-prisma.ts`

**Interface:**
```typescript
export function createTenantPrisma(tenantId: string): TenantScopedPrismaClient
```

**How it works:**
```typescript
export function createTenantPrisma(tenantId: string) {
  return prisma.$extends({
    query: {
      $allModels: {
        async $allOperations({ model, operation, args, query }) {
          const TENANT_MODELS = ['provider', 'providershift', 'providerabsence',
                                  'service', 'customer', 'appointment'];
          if (!TENANT_MODELS.includes(model.toLowerCase())) return query(args);

          if (['create', 'createMany'].includes(operation)) {
            args.data = Array.isArray(args.data)
              ? args.data.map((d: object) => ({ ...d, tenantId }))
              : { ...args.data, tenantId };
          }
          if (['findFirst','findMany','findUnique','findUniqueOrThrow',
               'count','aggregate','groupBy','update','updateMany',
               'delete','deleteMany','upsert'].includes(operation)) {
            args.where = { ...args.where, tenantId };
          }
          return query(args);
        },
      },
    },
  });
}
```

**Critical:** The `Tenant` model itself is NOT in `TENANT_MODELS` — tenant resolution queries the global prisma singleton, not the scoped client.

### 4. Tenant Middleware

**Purpose:** Resolve tenant from header, attach scoped Prisma client to request.

**Location:** `packages/api/src/http/middleware/tenant.middleware.ts`

**Interface:**
```typescript
export async function tenantMiddleware(
  request: FastifyRequest,
  reply: FastifyReply
): Promise<void>
```

**Attaches to request:**
```typescript
request.tenant: Tenant          // the resolved Tenant row
request.tenantPrisma: TenantScopedPrismaClient
```

**Registered in server.ts** as a global `preHandler` hook on all routes **except** `/api/platform/*` (super-admin routes use a different auth).

**Fastify type augmentation:**
```typescript
// src/types/fastify.d.ts
declare module 'fastify' {
  interface FastifyRequest {
    tenant: Tenant;
    tenantPrisma: TenantScopedPrismaClient;
    providerId?: string;  // renamed from barberId
  }
}
```

### 5. Repository Refactor (Prisma injection)

**Purpose:** Repos currently use the global Prisma singleton. They must accept a Prisma client in the constructor so tenant-scoped clients can be injected.

**Change pattern (all 5 repo files):**
```typescript
// Before:
export class PrismaAppointmentRepository {
  async create(data) { return prisma.appointment.create(...) }
}

// After:
export class PrismaAppointmentRepository {
  constructor(private db: PrismaClient | TenantScopedPrismaClient) {}
  async create(data) { return this.db.appointment.create(...) }
}
```

**Route files** instantiate repos per-request:
```typescript
// Before (module-level singleton):
const appointmentRepo = new PrismaAppointmentRepository();

// After (per-request, injected):
app.post('/book', async (request) => {
  const appointmentRepo = new PrismaAppointmentRepository(request.tenantPrisma);
  const useCase = new CreateAppointment(appointmentRepo, ...);
});
```

**Affected files:** `prisma-appointment.repository.ts`, `prisma-barber.repository.ts` (→ provider), `prisma-service.repository.ts`, `prisma-customer.repository.ts`, `prisma-barber-shift.repository.ts` (→ provider-shift)

### 6. Tenant-Aware Notification Service

**Purpose:** Replace hardcoded "Soberano Barbearia" with per-tenant config.

**Location:** `packages/api/src/infrastructure/notifications/whatsapp-notification.service.ts`

**Constructor change:**
```typescript
interface TenantNotificationConfig {
  businessName: string;
  providerLabel: string;
  bookingUrl: string;
}

export class WhatsAppNotificationService {
  constructor(
    private config: TenantNotificationConfig,
    private chatwoot: ChatwootClient,  // also becomes per-tenant
  ) {}
}
```

**`ChatwootClient` becomes per-tenant too** — Chatwoot credentials move from global env vars to `Tenant.config`:
```json
{
  "businessName": "Soberano Barbearia",
  "providerLabel": "Barbeiro",
  "bookingUrl": "https://soberano.altion.com.br",
  "chatwootBaseUrl": "https://...",
  "chatwootApiToken": "...",
  "chatwootAccountId": 1,
  "chatwootInboxId": 1
}
```

Routes instantiate the notification service with `request.tenant.config`.

**Global Chatwoot env vars removed** from `env.ts` and `docker-compose.yaml` — config lives in DB.

### 9. Frontend — `X-Tenant-Slug` Header Injection

**Purpose:** All API calls from any frontend must carry the tenant header.

**Location:** `packages/web/src/config/api.ts` + `packages/web/src/api/auth-request.ts`

```typescript
// New env var (build-time):
const TENANT_SLUG = import.meta.env.VITE_TENANT_SLUG;

// Added to every request's headers:
'X-Tenant-Slug': TENANT_SLUG,
```

**Both** fetch wrappers need this: `api` (public routes) and `authRequest` (admin routes).

**Vite env validation** — add to a `src/config/env.ts` in the web package:
```typescript
if (!import.meta.env.VITE_TENANT_SLUG) {
  throw new Error('VITE_TENANT_SLUG is required');
}
```

**No other changes** to the web frontend. Zero behavior change for Soberano.

### 11. `packages/web-admin` — Super-Admin Panel

**Purpose:** Private panel for Caio to create and manage tenants.

**Location:** `packages/web-admin/`

**Stack:** React 19 + Vite + TanStack Query + Tailwind — same as `packages/web`. Clone the package setup.

**Auth:** `SUPER_ADMIN_JWT_SECRET` env var (separate from barber JWT secret). Single admin user via env var credentials:
```
SUPER_ADMIN_EMAIL=caio@altion.com.br
SUPER_ADMIN_PASSWORD_HASH=bcrypt(...)
```
No `PlatformAdmin` DB model — avoids a whole auth system for one user.

**API routes** (new, under `/api/platform/`):
```
POST /api/platform/auth          → login, returns JWT
GET  /api/platform/tenants       → list all tenants
POST /api/platform/tenants       → create tenant
GET  /api/platform/tenants/:id   → get tenant
PATCH /api/platform/tenants/:id  → update tenant config / deactivate
```

**Frontend pages:**
```
/login          → Login form
/               → Tenant list (table: slug, name, type, active)
/tenants/new    → Create tenant form
/tenants/:id    → Edit tenant config (name, type, businessName, providerLabel, bookingUrl, Chatwoot credentials)
```

**Deployment:** `docker-compose.admin.yaml` — separate Coolify resource at `admin.altion.com.br`.

## Error Handling Strategy

| Scenario | Handler | Response |
|----------|---------|----------|
| Missing `X-Tenant-Slug` header | `tenantMiddleware` | `404 TENANT_NOT_FOUND` |
| Unknown tenant slug | `tenantMiddleware` | `404 TENANT_NOT_FOUND` |
| Tenant `isActive = false` | `tenantMiddleware` | `403 TENANT_INACTIVE` |
| Duplicate slug on create | platform route | `409 CONFLICT` |
| `Tenant.config` missing fields | Notification service | Fall back to defaults, no throw |
| Old JWT (no `tenantId`) | `authGuard` | `401 UNAUTHORIZED` — forces re-login |

## Tech Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Tenant resolution mechanism | `X-Tenant-Slug` header | Simpler than subdomain parsing; works for custom domains too; explicit |
| Auto-scoping approach | Prisma Client Extension `$allOperations` | Single injection point; no manual `tenantId` per query; prevents leaks |
| Repo instantiation | Per-request (inject scoped client) | Required for tenant context; aligns with existing use case injection pattern |
| Chatwoot config location | `Tenant.config` JSON | Each client has their own WhatsApp account; env vars can't support N tenants |
| Super-admin auth | Env var credentials + separate JWT secret | One user (Caio); avoids full auth system for MVP |
| API domain | Separate public `api.altion.com.br` | Enables independent frontend deployments; standard SaaS pattern |
| Barber→Provider in HTTP routes | Keep `/api/barbers` route paths | Zero breaking change for existing web frontend; internal rename only |
| Migration approach | 3 sequential migrations | Allows rollback at each step; production-safe |
